Build a real feature in the LINESETU Doctor App for **Walk-in Booking with Doctor Availability Date + Shift selection**, using:

--------------------------------------------------
CORE REQUIREMENT
--------------------------------------------------

Before adding a walk-in patient, doctor/receptionist must first select:

1. Doctor availability date
2. Shift (Morning / Evening)

This selected Date + Shift must use the **exact same queue logic** as the patient app e-token booking system.

That means:

- Every Date + Shift combination has its own unique queue
- Walk-in bookings and e-token bookings both enter the same queue
- Queue updates in real time
- Doctor app and patient app must always stay synced

Example:
- 15 Apr 2026 + Morning = Queue A
- 15 Apr 2026 + Evening = Queue B
- 16 Apr 2026 + Morning = Queue C

All of these are separate queues.

- Doctor can then add walk-in patients into that same selected queue

--------------------------------------------------
PATIENT APP SYNC REQUIREMENT
--------------------------------------------------

- Walk-in token from doctor app
- E-token booking from patient app

Do not create separate queue logic for walk-in and online.
Only one unified queue.

--------------------------------------------------
FIRESTORE DATA MODEL
--------------------------------------------------

Use a queue key based on:

- clinicId
- date
- shift

Recommended derived key:
queueKey = `${clinicId}_${yyyy-mm-dd}_${shift}`

Example:
clinicA_2026-04-15_morning

--------------------------------------------------
COLLECTIONS
--------------------------------------------------

1. clinics/{clinicId}
Fields:
- name
- doctorId
- availabilityDates: array or subcollection
- shiftsEnabled: { morning: true, evening: true }

2. bookings/{bookingId}
Fields:
- clinicId
- doctorId
- userId (null allowed for walk-in if needed)
- queueKey
- bookingDate (YYYY-MM-DD)
- shift ("morning" | "evening")
- tokenNumber
- type ("walk-in" | "online" | "emergency")
- status ("waiting" | "up_next" | "in_consultation" | "completed" | "skipped")
- name
- age
- gender
- phone
- createdAt
- source ("doctor_app" | "patient_app")

3. queueCounters/{queueKey}
Fields:
- clinicId
- doctorId
- bookingDate
- shift
- lastTokenNumber
- currentInConsultationToken
- currentUpNextToken
- updatedAt

--------------------------------------------------
TOKEN GENERATION LOGIC
--------------------------------------------------

When a new booking is created, whether from patient app or doctor app:

1. Identify queueKey from clinicId + date + shift
2. Use Firestore transaction
3. Increment queueCounters/{queueKey}.lastTokenNumber
4. Assign that as tokenNumber
5. Create booking inside same queueKey

This ensures:
- token numbers are unique per Date + Shift
- Morning and Evening are separate
- Every date has independent queue reset automatically by queueKey design

No manual reset needed if queueKey is date-based.
